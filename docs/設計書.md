# SCMリスク監視 業務設計書

Xトレンド監視 → カテゴリ分類 → ファクトチェック → 対応アクション

---

## 1. 業務フロー全体像

### ■ 業務フロー（5ステップ）

#### Step 1: トレンド取得

- 担当/手段: X API（自動処理）
- X API で定期的にトレンド・キーワードヒットを取得
  - Filtered Stream（常時接続）でリアルタイム取得
  - Search Recent API で5分間隔ポーリング補完

#### Step 2: カテゴリ分類＋フィルタ

- 担当/手段: AWS（自動処理）
- 取得データを事前定義カテゴリに自動分類
  - 第1段階: X APIクエリで無関係を除外
  - 第2段階: キーワードマッチで拠点×リスク照合
  - 第3段階: AI(Comprehend/Bedrock)で文脈判定
  - → 無関係と判定されたものは通知しない

#### Step 3: ファクトチェック

- 担当/手段: AWS(自動)＋人(判断)
- カテゴリ分類で「要確認」と判定されたトレンドについて
  - 主要ニュースサイト自動検索（NHK/共同通信/気象庁等）
  - 公式情報ソースとの照合
  - 信頼度スコア付与（SNS単独/ニュース裏付けあり/公式発表あり）

#### Step 4: リスク評価＋通知

- 担当/手段: AWS(自動)＋SCM担当
- 信頼度・影響度・緊急度を組合わせてリスクレベル判定
  - Level 3（重大）: 即時電話＋メール＋Slackなど
  - Level 2（警戒）: メール＋Slackなど
  - Level 1（注意）: Slackのみ / ダッシュボード反映

#### Step 5: 対応アクション

- 担当/手段: SCM部門＋関連部門
- リスクレベルに応じた標準対応手順を実行
  - 拠点への連絡・状況確認
  - 代替輸送/代替調達の検討・発動
  - 関連部門（生産管理・購買・物流）への共有
  - 経営層へのエスカレーション（Level 3）

## 2. キーワード設計

### X API キーワード設計

何を・どのように監視するかの具体定義

### ■ 2軸の掛け合わせで監視する

監視クエリ = 【リスクキーワード】 AND 【拠点キーワード】

例: `(地震 OR 火災 OR 浸水) AND (豊田市 OR 東名高速)`
→ これにより「自社SCMに関係のある情報だけ」を抽出する

### ■ 軸1: リスクキーワード（何が起きたか）

#### 1. 地震・津波

- 主要キーワード: 地震, 震度, 揺れ, 津波, 津波警報, 震源
- X API クエリ例: `(地震 OR 震度 OR 津波) lang:ja -is:retweet`
- 補足: 気象庁APIとの併用推奨

#### 2. 風水害

- 主要キーワード: 台風, 豪雨, 洪水, 浸水, 冠水, 土砂崩れ, 氾濫, 暴風
- X API クエリ例: `(台風 OR 豪雨 OR 洪水 OR 浸水 OR 土砂崩れ) lang:ja -is:retweet`
- 補足: 台風シーズンは監視強化

#### 3. 火災・爆発

- 主要キーワード: 火災, 火事, 黒煙, 爆発, 炎上（※物理的な）, 消防
- X API クエリ例: `(火災 OR 火事 OR 黒煙 OR 爆発) lang:ja -is:retweet -(炎上商法 OR 炎上案件)`
- 補足: 「炎上」はSNS炎上を除外

#### 4. 交通障害

- 主要キーワード: 通行止め, 道路封鎖, 渋滞, 事故, 不通, 運休, 遅延, 脱線, 立往生
- X API クエリ例: `(通行止め OR 不通 OR 運休 OR 脱線) lang:ja`
- 補足: 「渋滞」単独は日常的すぎるので他条件とのANDで使用

#### 5. 停電・インフラ障害

- 主要キーワード: 停電, 断水, ガス漏れ, 通信障害, 回線障害
- X API クエリ例: `(停電 OR 断水 OR 通信障害) lang:ja`
- 補足: 大規模停電は工場稼働に直結

#### 6. 労務・操業リスク

- 主要キーワード: ストライキ, 操業停止, 工場閉鎖, 倒産, 民事再生
- X API クエリ例: `(ストライキ OR 操業停止 OR 工場閉鎖 OR 倒産) lang:ja`
- 補足: サプライヤーの経営リスク監視

#### 7. 地政学・貿易

- 主要キーワード: 輸出規制, 制裁, 関税引上げ, 禁輸, 港湾封鎖, 海峡封鎖
- X API クエリ例: `(輸出規制 OR 制裁 OR 関税 OR 禁輸) (lang:ja OR lang:en)`
- 補足: 英語圏の一次情報も取得

#### 8. 感染症

- 主要キーワード: 感染拡大, パンデミック, 工場ロックダウン, 港湾閉鎖
- X API クエリ例: `(感染拡大 OR パンデミック OR ロックダウン) lang:ja`
- 補足: 流行期に監視強化

### ■ 軸2: 拠点キーワード（どこで起きたか）

#### 1. 自社工場

- 登録する地名・施設名: 工場名, 所在市区町村, 最寄り地名
  - 例: 〇〇工場, 豊田市, 高岡町
- 関連交通インフラ: 最寄りIC, 幹線道路, 鉄道路線
  - 例: 豊田IC, 東名高速, 伊勢湾岸道, 名鉄三河線
- 備考: 最重要拠点は周辺河川名も登録

#### 2. 自社倉庫

- 登録する地名・施設名: 倉庫名, 所在市区町村
  - 例: △△物流センター, 市川市
- 関連交通インフラ: 首都高湾岸線, 京葉道路, 国道357号, JR京葉線
- 備考: 物流拠点はIC/港湾のアクセス路も重要

#### 3. サプライヤー工場

- 登録する地名・施設名: サプライヤー名, 所在市区町村
  - 例: □□製作所, 堺市
- 関連交通インフラ: 阪神高速, 阪和自動車道, 南海本線
- 備考: Tier1は必須、Tier2は主要品目のみ

#### 4. 港湾

- 登録する地名・施設名: 港湾名, 所在地
  - 例: 名古屋港, 横浜港, 神戸港
- 関連交通インフラ: 周辺高速, コンテナターミナル名, 航路名
- 備考: 輸入原材料の入荷拠点を網羅

#### 5. 空港

- 登録する地名・施設名: 空港名
  - 例: 中部国際空港, 成田空港
- 関連交通インフラ: アクセス道路, 鉄道路線
- 備考: 航空輸送品目がある場合

#### 6. 幹線道路（拠点横断）

- 登録する地名・施設名: 東名高速, 新東名, 中央道, 名神, 東北道 等
- 関連交通インフラ: IC名, JCT名, SA/PA名
- 備考: 複数拠点を結ぶ路線は個別登録

### ■ 除外ルール（ノイズ排除）

| # | 除外対象 | 除外キーワード / ルール | X APIでの指定方法 | 理由 |
|---|----------|----------------------|------------------|------|
| 1 | リツイート | — | `-is:retweet` | 重複カウント防止、一次情報に限定 |
| 2 | Bot投稿 | — | `-from:bot_* -is:automated` | 自動投稿のノイズ排除 |
| 3 | エンタメ文脈 | ゲーム, アニメ, ドラマ, 映画, 小説, マンガ | `-(ゲーム OR アニメ OR ドラマ)` | 「爆発」「炎上」等の比喩的用法を排除 |
| 4 | 宣伝・広告 | — | `-has:links -is:promoted` | 広告ツイート排除（必要に応じて） |
| 5 | SNS炎上 | 炎上商法, 炎上案件, ネット炎上 | `-(炎上商法 OR 炎上案件 OR ネット炎上)` | 物理的な火災と区別 |

### ■ トレンド（バースト）検知

キーワード個別監視に加え、拠点地名の「ツイート量の急増」を検知する。
直近1時間のツイート数が過去7日平均の3倍を超えた場合にバーストと判定。
→ 事前想定していないリスク（未知のリスク）にも対応可能。

## 3. カテゴリ分類ロジック

取得したツイートをどのように分類し、無関係を除外するか

### ■ 3段階フィルタリング（詳細）

#### 第1段階（API層）: X APIクエリフィルタ

- 処理タイミング: X APIへのリクエスト時
- ロジック:
  - Filtered Stream に登録したルール（最大25ルール）に合致するツイートのみ取得
  - Search Recent API のクエリパラメータで絞込み
  - 除外キーワード（`-is:retweet`, `-(ゲーム OR アニメ)` 等）で事前除外
- 判定: 合致 → 次段階へ ／ 非合致 → そもそも取得しない
- 除外率目安: 全ツイートの95%以上を除外

#### 第2段階（マッチング層）: キーワードマッチング

- 処理タイミング: 取得後、即時（Lambda内）
- ロジック:
  - DynamoDB上の「拠点マスタ」と照合
  - リスクキーワード × 拠点キーワード のAND条件を確認
  - 合致した拠点ID・カテゴリIDをメタデータとして付与
  - どの拠点にもマッチしない → 除外（ただしバースト検知は別経路）
- 判定: マッチ → カテゴリ・拠点タグ付与 ／ 非マッチ → 除外
- 除外率目安: 残りの60〜70%を除外

#### 第3段階（AI層）: AI文脈判定

- 処理タイミング: 第2段階通過後
- ロジック:
  - Amazon Comprehend で感情分析（Negative/Mixed → リスク可能性高）
  - カスタム分類モデルで「真のリスク / 冗談・比喩 / 無関係」を判定
  - 例: 「豊田市で地震あった！怖い」→ リスク
  - 例: 「豊田市の地震保険CMうざい」→ 無関係
  - 信頼度スコア (0〜100) を付与
- 判定: スコア50以上 → 有効 ／ 50未満 → 除外
- 除外率目安: 残りの30〜50%を除外

### ■ 最終分類ラベル（出力）

| # | ラベル | 定義 | 付与条件 | 後続アクション |
|---|--------|------|----------|---------------|
| 1 | 確定リスク | ニュース等で裏付けが取れた実在するリスク事象 | AI信頼度80以上かつファクトチェック合格 | リスク評価→通知→対応 |
| 2 | 要確認 | SNS情報はあるがニュース未確認、事実確認が必要 | AI信頼度50〜79またはファクトチェック未完了 | ファクトチェック→判断待ち |
| 3 | 監視継続 | リスクの兆候はあるが現時点で影響は限定的 | AI信頼度50以上だが影響度が低と判定 | ダッシュボード表示、推移を監視 |
| 4 | 除外 | 無関係・冗談・比喩・誤検知 | AI信頼度50未満または除外ルール合致 | 記録のみ（精度改善用） |

## 4. ファクトチェック設計

トレンド検知後、事実確認をどのように行うか

### ■ ファクトチェック手順（自動＋人的確認）

#### Step 1: 公式API照合（自動）

- 所要時間: 数秒（自動）
- 処理内容:
  - 気象庁API: 地震・津波・台風の公式情報を自動照合
  - NEXCO/道路交通情報: 通行止め・規制情報の自動取得
  - 停電情報API: 各電力会社の停電情報取得
  - → 公式発表があれば「確定リスク」に即昇格
- 使用するソース/API:
  - 気象庁 XML/JSON
  - JARTIC 道路交通情報
  - 各電力会社 停電情報API

#### Step 2: ニュース検索（自動）

- 所要時間: 10〜30秒（自動）
- 処理内容:
  - Google News API / NewsAPI.org でキーワード検索
  - 主要ニュースソースに限定: NHK, 共同通信, 時事通信, 日経, 読売, 朝日, 毎日
  - → 1件以上ヒット: 信頼度+30
  - → 3件以上ヒット: 信頼度+50（ほぼ確定）
- 使用するソース/API:
  - Google News RSS
  - NewsAPI.org（Lambda定期実行）

#### Step 3: 公式SNS照合（自動）

- 所要時間: 数秒（自動）
- 処理内容:
  - 自治体・消防・警察・交通機関の公式Xアカウントを監視リストに登録
  - 例: @UN_NERV, @taborogawa, @ABOROGAWA, 各自治体防災アカウント
  - → 公式アカウントからの投稿は信頼度+40
- 使用するソース/API:
  - X API（公式アカウントリスト）

#### Step 4: 担当者による最終確認（人的）

- 所要時間: 1〜5分（人的判断）
- 処理内容:
  - 自動ファクトチェックで「要確認」のまま残ったもの:
  - 担当者がSlack通知で受け取り、5分以内に以下を確認
    - 元ツイートの内容・投稿者の信頼性
    - Google検索での追加情報確認
    - 必要に応じて拠点への電話確認
  - → 「確定リスク」「監視継続」「除外」のいずれかに最終判定
- 使用するソース:
  - Slack通知
  - 管理画面
  - 電話

### ■ ファクトチェック具体例

#### シナリオ1: 豊田市で震度5

- X上の情報: 「豊田市めっちゃ揺れた！」「震度5だって！工場大丈夫？」→ 短時間に50件超のツイート
- ファクトチェック結果: 気象庁API: 震度5弱を確認 → 合致、NHKニュース: 速報あり → 自動で「確定リスク」判定
- 信頼度: 100
- 最終判定: 確定リスク → 即時アラート

#### シナリオ2: 東名高速で事故渋滞

- X上の情報: 「東名高速で事故渋滞ひどすぎる」→ ツイート10件程度
- ファクトチェック結果: JARTIC: 東名高速 〇〇IC付近事故通行止め確認 → 自動で「確定リスク」判定
- 信頼度: 85
- 最終判定: 確定リスク → 物流影響確認

#### シナリオ3: 堺市の工場で黒煙目撃

- X上の情報: 「堺の方角から黒煙見える」→ ツイート3件
- ファクトチェック結果: 気象庁: 該当なし、ニュース: ヒットなし、公式SNS: 情報なし → 「要確認」のまま担当者へ
- 信頼度: 55
- 最終判定: 要確認 → 担当者が拠点に電話確認

#### シナリオ4: ゲームの地震イベント

- X上の情報: 「大地震来た！俺の街壊滅www」→ ゲームスレッドの文脈
- ファクトチェック結果: AI第3段階で「ゲーム文脈」と判定 → リスクスコア20で除外
- 信頼度: 20
- 最終判定: 除外
